# Requirements Document

## Introduction

The a2a-wrapper project currently consists of two independently-published npm packages (`a2a-copilot` and `a2a-opencode`) and a newly created shared core package (`@a2a-wrapper/core`). Each wrapper package has its own `.git` directory, CI/CD workflows, and duplicated community documentation. This feature restructures the project into a proper monorepo using npm workspaces and Turborepo for task orchestration, with a single git history, unified CI/CD, consolidated root-level documentation, flexible build targets, and independent package publishing — all without breaking existing npm consumers. The architecture must make it trivial to add new wrapper projects (e.g. `a2a-claude`) by implementing the core's `A2AExecutor` interface.

## Glossary

- **Monorepo**: A single Git repository containing multiple independently-versioned npm packages managed via npm workspaces
- **Root_Package_JSON**: The top-level `package.json` at the repository root that declares npm workspace paths
- **Workspace_Package**: Any npm package listed in the `workspaces` field of the Root_Package_JSON (`@a2a-wrapper/core`, `a2a-copilot`, `a2a-opencode`)
- **Community_Docs**: Repository-wide governance files — CONTRIBUTING.md, CODE_OF_CONDUCT.md, SECURITY.md, and LICENSE
- **CI_Workflow**: A GitHub Actions workflow that builds, type-checks, and tests code on push and pull request events
- **Publish_Workflow**: A GitHub Actions workflow that publishes changed packages to the npm registry on release events
- **Root_Gitignore**: The single `.gitignore` file at the repository root that applies to all Workspace_Packages
- **Package_Lock**: The root-level `package-lock.json` generated by npm workspaces that replaces per-package lock files
- **Turborepo**: A high-performance build system for JavaScript/TypeScript monorepos that provides task caching, dependency-aware task ordering, and parallel execution
- **Turbo_Pipeline**: The `turbo.json` configuration that defines task dependencies (e.g. `build` depends on `^build` of upstream packages) and caching rules
- **Changesets**: A versioning and changelog management tool for monorepos that tracks which packages have changed and automates version bumps and npm publishing
- **Changeset_PR**: An automated pull request created by the Changesets GitHub Action that batches pending version bumps and changelog entries for review before publishing

## Requirements

### Requirement 1: Root Package JSON with npm Workspaces

**User Story:** As a developer, I want a root-level `package.json` that declares all packages as npm workspaces, so that I can install, build, and test the entire monorepo with a single `npm install` command.

#### Acceptance Criteria

1. THE Root_Package_JSON SHALL declare a `workspaces` field using glob patterns (e.g. `["packages/*", "a2a-*"]`) that automatically include `a2a-copilot`, `a2a-opencode`, `packages/core`, and any future `a2a-*` wrapper directories
2. THE Root_Package_JSON SHALL set `private` to `true` to prevent accidental publishing of the root package
3. WHEN a developer runs `npm install` at the repository root, THE Root_Package_JSON SHALL cause npm to install dependencies for all Workspace_Packages and hoist shared dependencies
4. THE Root_Package_JSON SHALL include scripts that delegate to Turborepo (e.g. `"build": "turbo run build"`, `"test": "turbo run test"`, `"typecheck": "turbo run typecheck"`)
5. THE Root_Package_JSON SHALL set the `name` field to `a2a-wrapper`
6. THE Root_Package_JSON SHALL set `engines.node` to `>=18.0.0`
7. THE Root_Package_JSON SHALL include `turbo` and `@changesets/cli` as root-level devDependencies

### Requirement 2: Preserve Existing npm Package Names and Directory Layout

**User Story:** As an npm consumer of `a2a-copilot` or `a2a-opencode`, I want the package names and public API to remain unchanged after the monorepo restructure, so that my existing installations and imports continue to work without modification.

#### Acceptance Criteria

1. THE Workspace_Package `a2a-copilot` SHALL retain the npm package name `a2a-copilot` in its `package.json`
2. THE Workspace_Package `a2a-opencode` SHALL retain the npm package name `a2a-opencode` in its `package.json`
3. THE Workspace_Package `@a2a-wrapper/core` SHALL retain the npm package name `@a2a-wrapper/core` in its `package.json`
4. THE Monorepo SHALL keep the `a2a-copilot/` directory at the repository root without renaming or moving it to `packages/`
5. THE Monorepo SHALL keep the `a2a-opencode/` directory at the repository root without renaming or moving it to `packages/`
6. THE Monorepo SHALL keep the `packages/core/` directory at its current path
7. WHEN a consumer installs `a2a-copilot` from npm, THE published package SHALL contain the same `dist/`, `agents/`, and `LICENSE` files as before the restructure
8. WHEN a consumer installs `a2a-opencode` from npm, THE published package SHALL contain the same `dist/`, `agents/`, and `LICENSE` files as before the restructure

### Requirement 3: Consolidate Community Documentation at Root

**User Story:** As a contributor, I want a single set of community governance documents at the repository root, so that policies are consistent across all packages and maintained in one place.

#### Acceptance Criteria

1. THE Monorepo SHALL contain exactly one `CONTRIBUTING.md` file at the repository root
2. THE Monorepo SHALL contain exactly one `CODE_OF_CONDUCT.md` file at the repository root
3. THE Monorepo SHALL contain exactly one `SECURITY.md` file at the repository root
4. THE Monorepo SHALL contain exactly one `LICENSE` file at the repository root
5. WHEN the Community_Docs are consolidated, THE Monorepo SHALL remove `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, and `LICENSE` from the `a2a-copilot/` directory
6. WHEN the Community_Docs are consolidated, THE Monorepo SHALL remove `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, and `LICENSE` from the `a2a-opencode/` directory
7. THE root `CONTRIBUTING.md` SHALL reference the monorepo structure and provide instructions for working across all Workspace_Packages
8. THE root `SECURITY.md` SHALL cover security considerations for all Workspace_Packages in the Monorepo
9. EACH Workspace_Package SHALL retain its own `README.md` with package-specific documentation
10. EACH Workspace_Package SHALL retain its own `CHANGELOG.md` with package-specific release history

### Requirement 4: Preserve Package-Specific Documentation

**User Story:** As a developer evaluating a specific wrapper, I want each package to have its own README and changelog, so that I can understand that package's purpose, usage, and version history independently.

#### Acceptance Criteria

1. THE Workspace_Package `a2a-copilot` SHALL retain its own `README.md` in the `a2a-copilot/` directory
2. THE Workspace_Package `a2a-copilot` SHALL retain its own `CHANGELOG.md` in the `a2a-copilot/` directory
3. THE Workspace_Package `a2a-opencode` SHALL retain its own `README.md` in the `a2a-opencode/` directory
4. THE Workspace_Package `a2a-opencode` SHALL retain its own `CHANGELOG.md` in the `a2a-opencode/` directory
5. THE Workspace_Package `@a2a-wrapper/core` SHALL retain its own `README.md` in the `packages/core/` directory
6. THE Workspace_Package `@a2a-wrapper/core` SHALL retain its own `CHANGELOG.md` in the `packages/core/` directory

### Requirement 5: Unified CI/CD GitHub Actions Workflows

**User Story:** As a maintainer, I want a single set of GitHub Actions workflows at the repository root that use Turborepo for builds/tests and Changesets for publishing, so that CI/CD is centrally managed, cached, and handles granular releases automatically.

#### Acceptance Criteria

1. THE Monorepo SHALL contain a CI_Workflow at `.github/workflows/ci.yml` that runs on push to `main` and on pull requests to `main`
2. THE CI_Workflow SHALL use `turbo run build test typecheck` to build, type-check, and test all Workspace_Packages with correct dependency ordering and caching
3. THE CI_Workflow SHALL test against Node.js versions 18.x, 20.x, and 22.x
4. THE CI_Workflow SHALL use `npm ci` at the repository root to install dependencies for all Workspace_Packages
5. THE Monorepo SHALL contain a Publish_Workflow at `.github/workflows/publish.yml` that uses the Changesets GitHub Action
6. THE Publish_Workflow SHALL create a Changeset_PR when pending changesets exist, and publish to npm when the Changeset_PR is merged
7. THE Publish_Workflow SHALL use npm provenance for published packages
8. WHEN the unified workflows are created, THE Monorepo SHALL remove the `.github/` directory from `a2a-copilot/`
9. WHEN the unified workflows are created, THE Monorepo SHALL remove the `.github/` directory from `a2a-opencode/`

### Requirement 6: Unified Git Repository

**User Story:** As a developer, I want a single git history for the entire monorepo, so that cross-package changes can be tracked and reviewed in a single pull request.

#### Acceptance Criteria

1. WHEN the restructure is complete, THE Monorepo SHALL have a single `.git/` directory at the repository root
2. THE Monorepo SHALL remove the `.git/` directory from `a2a-copilot/`
3. THE Monorepo SHALL remove the `.git/` directory from `a2a-opencode/`
4. THE Monorepo SHALL contain a single Root_Gitignore at the repository root
5. THE Root_Gitignore SHALL combine the ignore patterns from both `a2a-copilot/.gitignore` and `a2a-opencode/.gitignore` without duplication
6. WHEN the Root_Gitignore is created, THE Monorepo SHALL remove `.gitignore` from `a2a-copilot/`
7. WHEN the Root_Gitignore is created, THE Monorepo SHALL remove `.gitignore` from `a2a-opencode/`

### Requirement 7: Consolidated Dependency Management

**User Story:** As a developer, I want a single `npm install` at the root to resolve and hoist all dependencies, so that shared dependencies are deduplicated and lock file conflicts are eliminated.

#### Acceptance Criteria

1. THE Monorepo SHALL contain a single Package_Lock at the repository root
2. WHEN the root Package_Lock is established, THE Monorepo SHALL remove `package-lock.json` from `a2a-copilot/`
3. WHEN the root Package_Lock is established, THE Monorepo SHALL remove `package-lock.json` from `a2a-opencode/`
4. WHEN the root Package_Lock is established, THE Monorepo SHALL remove `package-lock.json` from `packages/core/`
5. WHEN a developer runs `npm install` at the repository root, THE Monorepo SHALL install dependencies for all Workspace_Packages and create a single `node_modules/` at the root with hoisted shared dependencies
6. EACH Workspace_Package SHALL declare its own `dependencies` and `devDependencies` in its respective `package.json`

### Requirement 8: Independent Package Publishing

**User Story:** As a maintainer, I want each package to be publishable independently with its own version, so that a patch to `a2a-copilot` does not require a release of `a2a-opencode` or `@a2a-wrapper/core`.

#### Acceptance Criteria

1. EACH Workspace_Package SHALL maintain its own `version` field in its `package.json`
2. EACH Workspace_Package SHALL have `publishConfig.access` set to `public` in its `package.json`
3. THE Publish_Workflow SHALL support publishing any subset of the three Workspace_Packages in a single release cycle
4. WHEN `@a2a-wrapper/core` is published, THE published package SHALL include only the files specified in its `files` field (`dist/`)
5. WHEN `a2a-copilot` is published, THE published package SHALL include only the files specified in its `files` field (`dist/`, `agents/`, `LICENSE`)
6. WHEN `a2a-opencode` is published, THE published package SHALL include only the files specified in its `files` field (`dist/`, `agents/`, `LICENSE`)

### Requirement 9: Remove Per-Package Duplicate Artifacts

**User Story:** As a maintainer, I want all per-package duplicates (CI workflows, community docs, git directories, lock files, gitignore files) removed after consolidation, so that the repository has a single source of truth for each concern.

#### Acceptance Criteria

1. WHEN the restructure is complete, THE `a2a-copilot/` directory SHALL NOT contain any of: `.git/`, `.github/`, `.gitignore`, `package-lock.json`, `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `LICENSE`
2. WHEN the restructure is complete, THE `a2a-opencode/` directory SHALL NOT contain any of: `.git/`, `.github/`, `.gitignore`, `package-lock.json`, `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `LICENSE`
3. WHEN the restructure is complete, THE `packages/core/` directory SHALL NOT contain `package-lock.json`

### Requirement 10: Turborepo Task Orchestration

**User Story:** As a developer, I want Turborepo to manage build ordering, caching, and parallel execution, so that builds are fast, dependency-aware, and I don't need to manually sequence package builds.

#### Acceptance Criteria

1. THE Monorepo SHALL include `turbo` as a root-level devDependency
2. THE Monorepo SHALL contain a `turbo.json` configuration file at the repository root
3. THE Turbo_Pipeline SHALL define a `build` task that depends on `^build` (upstream package builds complete first), ensuring `@a2a-wrapper/core` is built before wrapper packages
4. THE Turbo_Pipeline SHALL define `test`, `typecheck`, and `clean` tasks
5. THE Turbo_Pipeline SHALL enable output caching for `build` and `typecheck` tasks so that unchanged packages are not rebuilt
6. THE Root_Package_JSON SHALL include scripts that delegate to `turbo run` (e.g. `"build": "turbo run build"`, `"test": "turbo run test"`)
7. THE CI_Workflow SHALL use `turbo run` commands instead of manual workspace iteration
8. THE Root_Gitignore SHALL include `.turbo/` to exclude Turborepo cache directories from version control

### Requirement 11: Flexible Build Targets and Module Compatibility

**User Story:** As a library consumer, I want the core package to emit both ESM and CommonJS-compatible output with declaration files, so that any Node.js application — regardless of its module system — can use it.

#### Acceptance Criteria

1. THE Workspace_Package `@a2a-wrapper/core` SHALL emit ESM output (`.js` files with `"type": "module"`) as the primary entry point
2. THE Workspace_Package `@a2a-wrapper/core` SHALL emit TypeScript declaration files (`.d.ts`) and declaration maps (`.d.ts.map`) alongside compiled JavaScript
3. THE Workspace_Package `@a2a-wrapper/core` SHALL emit source maps (`.js.map`) for debugging
4. THE `tsconfig.json` for each Workspace_Package SHALL use `"module": "NodeNext"` and `"moduleResolution": "NodeNext"` for maximum Node.js compatibility
5. THE `tsconfig.json` for each Workspace_Package SHALL target `"ES2022"` or later, but SHALL NOT hardcode a target that prevents use in Node.js 18+
6. THE `package.json` `exports` field for `@a2a-wrapper/core` SHALL specify both `"types"` and `"import"` conditions so that TypeScript and runtime resolution both work correctly
7. WHEN a new wrapper project is added, THE wrapper SHALL be able to import from `@a2a-wrapper/core` using standard ESM `import` syntax without additional configuration

### Requirement 12: Extensibility for New Wrapper Projects

**User Story:** As a maintainer, I want adding a new wrapper (e.g. `a2a-claude`) to require only creating a new directory with a `package.json`, executor implementation, and config defaults — with no changes to the root configuration or core package.

#### Acceptance Criteria

1. WHEN a new wrapper directory is added to the repository, THE Root_Package_JSON `workspaces` field SHALL use a glob pattern (e.g. `["packages/*", "a2a-*"]`) so that new `a2a-*` directories are automatically included without editing the root `package.json`
2. THE Turbo_Pipeline SHALL automatically discover and build new Workspace_Packages without changes to `turbo.json`
3. THE CI_Workflow SHALL automatically test new Workspace_Packages without changes to the workflow file
4. THE Publish_Workflow SHALL automatically publish new Workspace_Packages when they have a version bump, without changes to the workflow file
5. THE root `README.md` SHALL document the steps required to add a new wrapper project (create directory, implement `A2AExecutor`, define config, call `createCli`)
6. THE `@a2a-wrapper/core` package SHALL require zero code changes when a new wrapper is added

### Requirement 13: Granular Versioning and Release Management with Changesets

**User Story:** As a maintainer, I want a versioning system that tracks which packages changed, automates version bumps and changelog entries, and supports releasing any combination of packages — including core-only releases, single-wrapper releases, and multi-package releases.

#### Acceptance Criteria

1. THE Monorepo SHALL use `@changesets/cli` for version management and changelog generation
2. THE Monorepo SHALL contain a `.changeset/config.json` configuration file at the repository root
3. WHEN a developer makes a change to `@a2a-wrapper/core` only, THE Changesets tool SHALL allow creating a changeset that bumps only `@a2a-wrapper/core` without affecting wrapper package versions
4. WHEN a developer makes a change to `a2a-opencode` only, THE Changesets tool SHALL allow creating a changeset that bumps only `a2a-opencode` without affecting `a2a-copilot` or `@a2a-wrapper/core`
5. WHEN a developer makes a change to `@a2a-wrapper/core` that affects wrapper packages, THE Changesets tool SHALL support bumping both the core and affected wrapper packages in a single changeset
6. THE Publish_Workflow SHALL use the Changesets GitHub Action to create a Changeset_PR that batches pending version bumps and changelog entries
7. WHEN the Changeset_PR is merged, THE Publish_Workflow SHALL automatically publish all packages with pending version bumps to npm
8. THE Changesets configuration SHALL use `"fixed"` grouping set to `false` (independent versioning) so that each package maintains its own version independently
9. EACH Workspace_Package's `CHANGELOG.md` SHALL be automatically updated by Changesets when a version is bumped
